\begin{abstract}
\noindent Persistent memory is a first-order requirement for multi-session agent workflows. This report documents why Cortex chose a markdown-native, git-backed architecture with hook-driven lifecycle automation, trust-scored retrieval, and configurable governance. The core argument is operational: memory quality must be optimized alongside auditability, governance, portability, and token predictability. Cortex requires no account, no API key, and no required hosted service in its default FTS5 path; the store bootstraps locally and retrieval runs against a local SQLite FTS5 index with no network hop, no token cost, and no rate limits. The main tradeoff is operational shape: managed alternatives reduce operator burden, while Cortex prioritizes portability, auditability, and ownership of the memory corpus. This report compares Cortex to Supermemory, claude-mem, and GitHub Copilot Memory, and separates \textit{architecture judgments} from \textit{measured performance}; sections without empirical data say so directly.
\end{abstract}

\section{Decision Criteria}
\label{sec:criteria}
Systems are evaluated on eight dimensions:
\begin{multicols}{2}
\begin{enumerate}
  \item \textbf{Representation}: where memory lives and who owns it.
  \item \textbf{Capture path}: how memory is created (hooks/API/workers).
  \item \textbf{Retrieval path}: ranking, trust scoring, and injection under budget.
  \item \textbf{Governance}: forgetting, validation, review, and policy controls.
  \item \textbf{Economics}: token overhead + human maintenance burden.
  \item \textbf{Portability}: migration risk and lock-in profile.
  \item \textbf{Multi-agent readiness}: concurrent sessions, shared state, coordination.
  \item \textbf{Operability}: CLI, shell, direct management without the model in the loop.
\end{enumerate}
\end{multicols}

\subsection{Storage and Retrieval}
Cortex keeps memory in project markdown files and builds a local SQLite FTS5 index for retrieval \cite{cortex-readme,cortex-shared,cortex-cli}. Markdown files remain the single source of truth; the index is a cached snapshot that hook and MCP entrypoints load and refresh incrementally. This yields strong inspectability (diffs, git history, plain-text grep) and low platform lock-in.

Before injection, a trust filter scores each entry by age decay and citation validity (Section~\ref{sec:decay}), suppressing entries below a configurable confidence floor. Archived entries are stripped so consolidated history does not pollute active search. When semantic recovery is enabled, embeddings are cached locally and init can pre-warm coverage so the first real session is not starting from zero. Recovery queries use a persistent candidate index rather than re-embedding the corpus on every search.

\subsection{Lifecycle Automation}
Three hook stages automate retrieval and persistence \cite{cortex-cli,cortex-hooks}:
\begin{enumerate}
  \item \texttt{SessionStart}: pulls the cortex repo and warms the index cache.
  \item \texttt{UserPromptSubmit}: extracts keywords, searches the index, and injects relevant snippets within the token budget. Fires a consolidation notice when thresholds are met.
  \item \texttt{Stop}: auto-commits local cortex changes after every response and schedules background sync when a remote exists.
\end{enumerate}

\subsection{Token Budget and Progressive Disclosure}
\label{sec:context-opt}
MCP tool responses consume context tokens. Cortex controls this with progressive disclosure: automatic injection surfaces compact summaries with stable \texttt{mem:} IDs for large result sets, and agents expand only what they need. Backlogs support summary-only, paginated, and single-item fetch modes. The design principle: fetch the minimum that answers the question.

\subsection{Governance}
\label{sec:governance}
The governance engine \cite{cortex-shared} provides four configurable policy layers: retention and TTL (default 120-day decay, 365-day retention), workflow approval gates for risky queue items, role-based access control (admin/maintainer/contributor/viewer with 6 action types), and canonical locks protecting pinned memories from bulk overwrites. All governance events are appended to a timestamped audit log.

\subsection{Trust Decay and Confidence Scoring}
\label{sec:decay}

Not all memories deserve equal weight. Each entry is scored before injection:
\begin{itemize}
  \item \textbf{Age decay}: confidence degrades over four windows (\texttt{d30}/\texttt{d60}/\texttt{d90}/\texttt{d120}), defaulting to 1.0, 0.85, 0.65, and 0.45.
  \item \textbf{Citation validity}: entries with valid file/line/commit citations score higher; entries without citations receive a 0.8 multiplier.
  \item \textbf{Minimum confidence}: entries below 0.35 are suppressed entirely.
  \item \textbf{Quality feedback}: helpful-count, reprompt penalty, and regression penalty are tracked per entry.
\end{itemize}

\subsection{Extensibility}
Seven user-defined hook events (\texttt{pre/post-save}, \texttt{post-search}, \texttt{pre/post-finding}, \texttt{pre/post-index}) support custom integrations \cite{cortex-hooks}. Projects export and import as self-contained JSON bundles. Migration tooling mines git history, CI failures, and PR reviews to seed an initial \texttt{FINDINGS.md}.

\section{Multi-Agent Coordination}
Modern agent workflows frequently involve multiple agents running concurrently. Memory systems that assume a single serial session break under this pattern.

Cortex handles multi-agent use through three mechanisms. \textbf{Shared git-backed state}: all agents read and write the same \texttt{\textasciitilde/.cortex} directory. Concurrent reads are safe at the file system layer; MCP writes are serialized within a process by file locks, with cross-process synchronization resolving at the git layer. \textbf{Skill-driven coordination}: skills in \texttt{global/skills/} define reusable agent behaviors injected per project. \textbf{Hook isolation}: each agent session gets its own execution context; the consolidation notice fires once per session via marker files.

\subsection{Operational Differentiators}

The systems differ less in whether they can retrieve a relevant memory and more in how they behave operationally once memory is a shared, long-lived artifact.

\begin{itemize}
  \item \textbf{Git-backed auditability}: every memory write is a commit. You can \texttt{git blame} any learning, revert a bulk operation, or audit who added what and when. Managed API systems rarely offer equivalent transparency.
  \item \textbf{Policy and access governance}: role-based access control, approval workflows, TTL and retention policy, and canonical locks are first-class features.
  \item \textbf{Trust filtering}: citation decay, confidence scores, and regression penalties suppress stale or low-quality memories before injection.
  \item \textbf{CLI shell for human review}: \texttt{cortex shell} provides a model-free interface for reviewing, approving, and cleaning memories. Particularly useful at team scale.
  \item \textbf{Browser-based memory triage}: \texttt{cortex review-ui} launches a lightweight web UI for reviewing the memory queue, filtering by machine, model, and project. No comparable interface surfaced in the evaluated materials.
\end{itemize}

Any memory system can add a vector index. Governance, auditability, and operational tooling require deliberate architectural choices that are harder to retrofit.

\section{FTS5 vs Semantic Search}
\label{sec:fts5}

Cortex is fundamentally anchored in SQLite FTS5 (BM25 ranking), with fallback layers in some paths. That choice is deliberate and has real consequences.

\textbf{FTS5 advantages:} zero required hosted dependencies in the default path; deterministic and auditable results; low-latency lookup on typical corpus sizes; works offline with no retrieval-side token cost or rate limits.

\textbf{FTS5 limitations:} vocabulary mismatch (``authentication'' won't surface ``login flow'' without synonym expansion); no semantic relatedness across paraphrased concepts; BM25 ranks by term frequency, not relevance.

\textbf{The honest tradeoff:} semantic search retrieves better on paraphrase and synonym-heavy queries. FTS5 retrieves better when vocabulary is consistent, offline operation is required, or zero retrieval-side token cost matters.

\subsection{Hybrid Retrieval: Current State}
\label{sec:hybrid}

Cortex now shares the same staged hybrid policy across hook injection and \texttt{search\_knowledge}: strict FTS5 first, a relaxed lexical rescue query if the strict pass misses, cheap local fallback scoring, and vector recovery only when lexical hits are sparse or weak. The vector path uses persistent embeddings and a cached candidate index, then the shared reranker weights local lexical overlap ahead of weaker semantic tails. Benchmark claims still require published conditions because corpus size, cache warmth, and query mix change the outcome \cite{cortex-bench}.

\subsection{Retrieval Complexity}

The staged design matters because each retrieval tier has a different cost profile. The practical hot path is sublinear because the linear fallbacks are gated and bounded rather than paid on every request.

\subsection{Measured Retrieval Notes}

An author-local legacy run on March 9, 2026 used 10 real queries against a live cortex store with 139 indexed documents and the default 550-token hook budget \cite{cortex-bench}. Under those conditions:
\begin{itemize}
  \item the lexical path averaged 15.93\,ms total latency (p50 11.91\,ms, p95 16.25\,ms);
  \item the gated semantic path averaged 12.90\,ms total latency (p50 11.59\,ms, p95 16.52\,ms);
  \item average injected payload was 341.8 tokens in both modes;
  \item both modes hit all 10 queries, and the two previously published miss cases were resolved by the lexical fixes;
  \item the persistent vector index reduced the average cosine-scoring candidate set to 8.7 candidates out of 139 eligible documents (6.3\% of the corpus).
\end{itemize}

A widened 16-query run on the same warm store averaged 14.71\,ms lexical latency and 13.60\,ms gated-semantic latency, with both modes again hitting every query and producing no semantic-only recoveries. The interpretation is important: on this corpus the strengthened lexical path is now good enough that the vector gate usually stays closed, so the hybrid path inherits the same speed profile. The vector index still matters for worst-case scaling because it shrinks the cosine stage to 8.6\% of the eligible corpus on the widened run, but when semantic recovery does fire the dominant cost is still query embedding, not cosine math.

\section{Limitations}
\label{sec:limitations}

\textbf{FTS5 retrieval ceiling.} Vocabulary mismatch remains a real failure mode in large projects, partially mitigated by synonym expansion.

\textbf{Git contention under concurrent writes.} Heavy multi-agent parallelism will produce merge conflicts or push failures at the git layer; in-process locks do not help cross-machine, and the system is eventually consistent rather than strongly coordinated.

\textbf{Hook-path indexing overhead.} Index load and validation adds perceptible latency for large cortex repos, even with a warm cache.

\textbf{Ranking quality.} BM25 is not calibrated to agent-context relevance. The quality feedback loop requires usage history; new projects start cold.

\textbf{Semantic recovery latency.} On small corpora, vector candidate pruning changes asymptotic behavior but does not automatically improve wall-clock latency. Query embedding remains the dominant cost when semantic recovery fires.

\textbf{Residual retrieval uncertainty.} The published March 9 benchmark set no longer has misses, but that is still bounded evidence for one corpus and one query set. Broader paraphrase-heavy evaluation is still needed before making stronger semantic-completeness claims.

\textbf{Governance complexity.} The four-file governance system is powerful but non-trivial to configure. Initial setup is guided, but advanced policy configuration still requires direct CLI commands.

\textbf{Review UI is opt-in.} \texttt{cortex review-ui} is not started automatically; memory triage requires an explicit invocation. This is intentional---triage is a deliberate act, not an ambient background process.

\textbf{Uni-directional portability.} Export/import works for project data; migrating governance configuration to another memory system is manual.

\section{Conclusion}
The decision is less about ``who retrieves best on a single prompt'' and more about long-horizon operational fit. Cortex pushes toward production readiness: trust filtering reduces noise in injected context, the governance engine gives teams real policy control, and the interactive shell makes daily memory management possible without starting an LLM session.

Multi-agent support is increasingly non-optional. As workflows move from single-agent to team coordination, memory systems that assume serial sessions become a bottleneck. Cortex's git-backed architecture provides a workable shared-memory baseline, with the acknowledged caveat that heavy concurrent writes create contention that requires care.

Managed alternatives are a reasonable choice for teams already fully committed to a specific platform where memory comes bundled and data custody is not a concern. Cortex is strongest when the priority is owner-operated memory: markdown as source of truth, predictable token discipline, and auditability through git. In practice, hooks auto-pull, auto-commit, and schedule maintenance around the workflow; consolidation itself is still surfaced explicitly when it is time to clean up findings. In the default FTS5 path there are no API costs, no rate limits, and no required hosted service, but there is still real operator responsibility around git sync, local setup, and occasional contention.
