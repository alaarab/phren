\begin{abstract}
Persistent memory is now a first-order requirement for multi-session coding workflows. This report documents why Cortex chose a markdown-native, git-backed architecture with hook-driven lifecycle automation, trust-scored retrieval, and configurable governance. Cortex v1.11.1 includes a policy engine (retention, decay, access control), an interactive shell for direct memory management, index-level filtering, and first-class support for multi-agent coordination. The report compares Cortex to Supermemory, claude-mem, and GitHub Copilot Memory, provides a cost model with worked examples, and proposes a reproducible benchmark protocol. The central argument is operational: in real engineering teams, memory quality must be optimized jointly with auditability, governance, portability, and token predictability.
\end{abstract}

\section{Executive Summary}
A memory system for coding agents must solve four coupled problems:
\begin{enumerate}
  \item preserve context over time,
  \item retrieve relevant context under token constraints,
  \item suppress stale or low-confidence memory,
  \item remain operable at team scale.
\end{enumerate}

Cortex optimizes for ownership and controllability. Alternatives can outperform on managed convenience, setup simplicity, or deep platform integration. The right choice depends on which constraints are binding.
This report separates \textit{architecture judgments} from \textit{measured performance}; they are not interchangeable. Sections without empirical data say so directly.

\section{Decision Criteria}
This report evaluates systems on eight dimensions:
\begin{enumerate}
  \item \textbf{Representation}: where memory lives and who owns it.
  \item \textbf{Capture path}: how memory is created (hooks/API/workers).
  \item \textbf{Retrieval path}: ranking, trust scoring, and injection under budget.
  \item \textbf{Governance}: forgetting, validation, review, and policy controls.
  \item \textbf{Economics}: token overhead + human maintenance burden.
  \item \textbf{Portability}: migration risk and lock-in profile.
  \item \textbf{Multi-agent readiness}: concurrent sessions, shared state, coordination.
  \item \textbf{Operability}: CLI, shell, direct management without the model in the loop.
\end{enumerate}

\subsection{Storage and Retrieval}
Cortex keeps memory in project markdown files and builds a local SQLite FTS5 index for retrieval \cite{cortex-readme,cortex-shared,cortex-cli}. The index is rebuilt on each session start from the file system, so markdown files remain the single source of truth. This yields strong inspectability (diffs, git history, plain-text grep) and low platform lock-in.

A configurable index policy \cite{cortex-shared,cortex-index} controls which files are indexed. Operators set \texttt{includeGlobs} (default: \texttt{**/*.md}), \texttt{excludeGlobs} (default: \texttt{node\_modules}, \texttt{.git}), and an \texttt{includeHidden} flag. The policy is hashed into the index cache key, so changing it forces a clean rebuild and prevents accidental indexing of vendor files or sensitive directories.

Before injection, a trust filter scores each learning entry by age decay and citation validity (Section~\ref{sec:decay}), then suppresses entries below \texttt{minInjectConfidence=0.35}. Archived entries inside \texttt{<details>} blocks are stripped so consolidated history does not pollute active search.

\subsection{Lifecycle Automation}
Three hook stages automate retrieval and persistence \cite{cortex-cli,cortex-hooks}:
\begin{enumerate}
  \item \texttt{SessionStart}: pulls the cortex repo, rebuilds the FTS5 index.
  \item \texttt{UserPromptSubmit}: extracts keywords, expands synonyms, searches the index, injects relevant snippets within the token budget ($\approx$550 tokens, 6 lines, 520 chars per snippet). Also checks consolidation thresholds and fires a one-per-session notice when the project has accumulated $\geq$25 new learnings.
  \item \texttt{Stop}: auto-commits and pushes all cortex changes after every response, including subagent responses.
\end{enumerate}
This reduces dependence on manual tool invocation and lowers workflow variance across sessions.

\subsection{Governance}
\label{sec:governance}

The governance engine lives in \texttt{.governance/} \cite{cortex-shared} and is composed of four policy files:

\textbf{Memory policy} (\texttt{memory-policy.json}): TTL (\texttt{ttlDays=120}), retention (\texttt{retentionDays=365}), auto-accept confidence threshold, minimum injection confidence, and the four-tier decay curve. Changes are audit-logged.

\textbf{Workflow policy} (\texttt{memory-workflow-policy.json}): approval gates controlling whether new memories require maintainer approval and which queue sections trigger review.

\textbf{Access control} (\texttt{access-control.json}): role-based permissions with four roles: \texttt{admin}, \texttt{maintainer}, \texttt{contributor}, \texttt{viewer}. Each maps to a set of allowed actions (\texttt{read}, \texttt{write}, \texttt{queue}, \texttt{pin}, \texttt{policy}, \texttt{delete}). The system resolves the current actor's role and rejects unauthorized operations before they execute.

\textbf{Canonical locks} (\texttt{canonical-locks.json}): pinned memories are protected by content hashes. This prevents accidental overwrite during consolidation or bulk operations.

All governance mutations are appended to \texttt{memory-usage.log} with timestamps, providing a complete, human-readable audit trail without requiring database queries.

\subsection{Trust Decay and Confidence Scoring}
\label{sec:decay}

Not all memories deserve equal weight. Each learning entry is scored before injection:
\begin{itemize}
  \item \textbf{Age decay}: confidence degrades over four configurable windows (\texttt{d30}/\texttt{d60}/\texttt{d90}/\texttt{d120}), defaulting to 1.0, 0.9, 0.75, and 0.5 respectively.
  \item \textbf{Citation validity}: entries with valid file/line/commit citations score higher. Entries without citations receive a 0.8 multiplier.
  \item \textbf{Minimum confidence}: entries below \texttt{minInjectConfidence=0.35} are suppressed.
  \item \textbf{Quality feedback}: impressions, helpful-count, reprompt penalty, and regression penalty are tracked per entry. Entries that cause regressions lose injection priority over time.
\end{itemize}

\subsection{Extensibility}

Beyond the three lifecycle hooks, Cortex supports user-defined hooks for seven events: \texttt{pre-save}, \texttt{post-save}, \texttt{post-search}, \texttt{pre-learning}, \texttt{post-learning}, \texttt{pre-index}, and \texttt{post-index} \cite{cortex-hooks}. Hooks receive \texttt{CORTEX\_PATH} and \texttt{CORTEX\_HOOK\_EVENT} as environment variables, plus event-specific data. Typical uses include Slack notifications on new learnings, webhook calls after index rebuilds, and pre-save validation scripts.

Projects can be exported as self-contained JSON bundles via \texttt{export\_project} and re-imported with \texttt{import\_project}. The \texttt{manage\_project} tool supports archiving inactive projects without deleting data. Migration tooling (\texttt{cortex maintain migrate <project>}) scans git history and GitHub PR reviews, CI failures, and issue comments, routing extracted candidates to \texttt{LEARNINGS.md} or \texttt{MEMORY\_QUEUE.md} based on confidence.

\section{Interactive Shell}
The terminal-native shell (\texttt{cortex shell}) provides direct access to memory operations without requiring an LLM in the loop \cite{cortex-cli}. Six views are navigable by single keystroke: Projects, Backlog, Learnings, Memory Queue, Machines/Profiles, and Health diagnostics.

A palette system (\texttt{:command}) supports structured operations: \texttt{:add}, \texttt{:complete}, \texttt{:move}, \texttt{:govern}, \texttt{:consolidate}, \texttt{:learn add}, \texttt{:mq approve}, and more. Filtering (\texttt{/keyword}) and pagination (\texttt{:page next}) handle large datasets. State persists across invocations (current view, project, page, filter) so engineers return to where they left off.

The design goal is operational: memory governance should not require starting a coding session. An engineer should be able to run \texttt{cortex shell}, review what accumulated overnight, and clean it up before any agent touches it.

\section{Multi-Agent Coordination}
Modern coding workflows frequently involve multiple agents running concurrently. Memory systems that assume a single serial session break under this pattern.

Cortex handles multi-agent use through three mechanisms. First, \textbf{shared git-backed state}: all agents read and write the same \texttt{\textasciitilde/.cortex} directory. Concurrent reads are safe at the file system layer. Write contention is managed by git: the Stop hook commits and pushes after every response, and SessionStart pulls at session start. Conflicts are resolved at the git layer, not in application code.

Second, \textbf{skill-driven coordination}: Cortex skills (markdown files in \texttt{global/skills/}) define reusable agent behaviors. A \texttt{/swarm} skill reads backlog items and spawns team agents that self-assign tasks, report back via messaging, and share learnings through the same cortex instance.

Third, \textbf{hook isolation}: each agent session gets its own hook execution context. The consolidation notice fires once per session (tracked via \texttt{.noticed-\{session\_id\}} files), preventing parallel agents from sending duplicate prompts. The Stop hook fires after every subagent response, so learnings from specialist agents are committed immediately.

\section{API vs MCP vs Local}
These concepts are often conflated:
\begin{itemize}
  \item \textbf{MCP} is an integration protocol (tooling interface).
  \item \textbf{Local vs API} is a storage/compute decision (where memory runs).
\end{itemize}

Cortex is \textit{local memory + MCP interface + lifecycle hooks}. A local system can still expose MCP tools. The 19 MCP tools cover search, backlog management, learning capture, memory quality feedback, and data portability \cite{cortex-index}.

\section{Speed and Cost Analysis}

\subsection{Latency decomposition}
End-to-end latency decomposes into five stages:
\begin{enumerate}
  \item retrieval/index lookup,
  \item ranking/trust filtering,
  \item prompt augmentation,
  \item model generation,
  \item post-response persistence and governance.
\end{enumerate}

\subsection{Operational cost equation}
\begin{align*}
\mathrm{Total\ Cost} &= \mathrm{Token\ Cost} + \mathrm{Infra\ Cost}\\
&\quad + (\mathrm{Engineer\ Hours} \times \mathrm{Loaded\ Rate})
\end{align*}

\begin{align*}
\mathrm{Token\ OpEx}_{month} &= N \times ((T_{in}+T_{inj}) \times p_{in}\\
&\quad + T_{out} \times p_{out})
\end{align*}

In production teams, the labor term is routinely underestimated. Governance cleanup and workflow drift often dominate once basic retrieval works.

\subsection{Worked cost example}

Consider a team running 500 prompts/month against Claude Opus 4.6 (input \$15/M tokens, output \$75/M tokens as of March 2026). Assume 1,500 tokens average input, 800 tokens average output, and Cortex injecting its default 550-token budget per prompt.

\begin{align*}
T_{in,total} &= 1{,}500 + 550 = 2{,}050 \text{ tokens/prompt}\\
\mathrm{Token\ Cost} &= 500 \times (2{,}050 \times \tfrac{15}{10^6} + 800 \times \tfrac{75}{10^6})\\
&= 500 \times (0.03075 + 0.060)\\
&= 500 \times 0.09075 \approx \$45.38/\mathrm{mo}
\end{align*}

Without memory injection (no Cortex overhead):
\begin{align*}
\mathrm{Token\ Cost}_{baseline} &= 500 \times (1{,}500 \times \tfrac{15}{10^6} + 800 \times \tfrac{75}{10^6})\\
&\approx \$41.25/\mathrm{mo}
\end{align*}

\textbf{Memory overhead: \$4.13/mo} at 500 prompts. At 5,000 prompts/month, the overhead is \$41.25 -- still modest relative to the value of reduced repetition and context re-entry. If Cortex prevents even two engineer-hours of re-explanation per month (at a loaded rate of \$100/hr), the economics strongly favor injection.

The labor and governance terms are harder to model but often larger. A team that spends 30 minutes/week on memory triage pays roughly \$200/mo in loaded time -- which argues for governance automation over manual review.

\section{FTS5 vs Semantic Search}
\label{sec:fts5}

This section addresses the retrieval tradeoff the paper has been sidestepping: Cortex uses SQLite FTS5 (keyword/BM25 ranking), not embedding-based semantic search. That is a deliberate choice with real consequences.

\textbf{FTS5 advantages:}
\begin{itemize}
  \item Zero external dependencies -- no embedding model, no vector DB, no API call.
  \item Deterministic and auditable: given the same query and index, you get the same results.
  \item Sub-millisecond lookup on typical cortex corpus sizes (hundreds to low thousands of documents).
  \item Works offline, no token cost for retrieval, no rate limits.
\end{itemize}

\textbf{FTS5 limitations:}
\begin{itemize}
  \item Vocabulary mismatch: a query about ``authentication'' won't surface a learning stored as ``login flow.'' Cortex partially mitigates this with a synonym expansion layer, but synonym coverage is hand-curated, not learned.
  \item No semantic relatedness: FTS5 cannot bridge ``the CI pipeline flakes on ARM'' to a learning about ``cross-compilation issues on Apple Silicon,'' even if they describe the same problem.
  \item Ranking quality: BM25 is term-frequency based. It will surface a document that mentions the query word ten times over a more relevant document that mentions it once.
\end{itemize}

\textbf{When the tradeoff bites:} retrieval quality degrades when the user's vocabulary and the stored learning's vocabulary diverge. This is more common in large, long-lived projects with many contributors. It is less common in single-developer projects where vocabulary is consistent.

\textbf{The honest comparison:} semantic search (e.g., Supermemory's embedding layer) genuinely retrieves better on paraphrase and synonym-heavy queries. Cortex's FTS5 approach retrieves better when the user knows the exact terminology, when offline operation or auditability are required, and when zero retrieval-side token cost matters.

A hybrid approach -- FTS5 for exact matches, embedding fallback for low-result queries -- would close most of the gap. This is not currently implemented.

\section{When To Choose What}
\subsection{Choose Cortex when}
\begin{itemize}
  \item memory ownership, auditability, and portability are hard constraints,
  \item you need explicit token-budget controls and trust-scored retrieval,
  \item multiple agents share a codebase and need concurrent memory access,
  \item governance matters: role-based access, approval workflows, audit trails,
  \item you want direct memory management through a terminal shell.
\end{itemize}

\subsection{Choose alternatives when}
\begin{itemize}
  \item managed API velocity and external connectors matter more than portability (Supermemory),
  \item deep GitHub-native convenience is primary (Copilot Memory),
  \item local custom hook pipelines with broader runtime composition are preferred (claude-mem),
  \item semantic retrieval quality across large, vocabularily diverse corpora is critical (Supermemory).
\end{itemize}

\section{Proposed Benchmark Protocol}
\label{sec:benchmark}

\textbf{Note: this is a proposed evaluation framework. No results have been collected against it yet. It is published here to define what a fair comparison would require, not to imply that Cortex has been measured against these criteria.}

A minimum viable benchmark uses:
\begin{itemize}
  \item paired task-level design across systems,
  \item fixed model/runtime controls,
  \item at least 40 tasks across multiple repositories,
  \item bootstrap confidence intervals for primary endpoints.
\end{itemize}

\subsection{Model matrix (practical default)}
Run each system with the same task set across:
\begin{itemize}
  \item Claude family: Opus 4.6, Sonnet 4.6, Haiku 4.6.
  \item OpenAI family: GPT-5.2, GPT-5.3, Codex.
\end{itemize}

Report both \textbf{memory-path latency} (retrieval, ranking, augmentation, persistence) and \textbf{end-to-end latency} (including generation) for each system/model combination.

\subsection{Primary endpoints}
\begin{itemize}
  \item task success rate,
  \item mean injected tokens,
  \item latency p95,
  \item citation validity rate.
\end{itemize}

\section{Limitations}
\label{sec:limitations}

Honest systems documentation acknowledges what doesn't work well. Here is what Cortex gets wrong or trades away:

\textbf{FTS5 retrieval ceiling.} As described in Section~\ref{sec:fts5}, keyword search misses semantically related content. In practice, synonym expansion helps on the most common cases, but vocabulary mismatch remains a real failure mode in large projects.

\textbf{Git contention under concurrent writes.} The Stop hook's commit-and-push design assumes low write concurrency. Under heavy multi-agent parallelism (many agents each finishing responses close together), you will get git merge conflicts or push failures. The current design resolves these at the git layer, which means at least one agent's write wins and the others need to retry. There is no application-level write serialization beyond the in-process queue for single-session concurrent tool calls.

\textbf{Cold-start latency.} The FTS5 index is rebuilt on SessionStart from the file system. For large cortex repos (thousands of markdown files), this adds perceptible startup latency. The index is cached and invalidated only on policy change or structural change, but the first build after install or policy change is not instant.

\textbf{Ranking quality.} BM25 is not calibrated to coding-context relevance. The quality feedback loop (helpful/reprompt/regression signals) partially compensates over time, but requires enough usage to build signal. New projects start with no feedback history and pure BM25 ranking, which may surface noisy results early on.

\textbf{Governance complexity.} The four-file governance system is powerful but non-trivial to configure correctly. Default settings are reasonable, but teams that need custom TTL, role granularity, or approval workflows will spend time on configuration. There is no guided setup beyond the CLI commands.

\textbf{No web-based review UI by default.} The \texttt{cortex memory-ui} command starts a local web UI, but it is not on by default. Memory triage is primarily a terminal workflow. Teams expecting a browser-based management interface will need to adapt.

\textbf{Portability is uni-directional for now.} Export/import works for project data, but migrating the governance configuration (policies, roles, canonical locks) to a different memory system is manual work. The format is not standardized across tools.

\section{Conclusion}
The decision is less about ``who retrieves best on a single prompt'' and more about long-horizon operational fit. Cortex pushes toward production readiness: trust filtering reduces noise in injected context, the governance engine gives teams real policy control, and the interactive shell makes daily memory management possible without starting an LLM session.

Multi-agent support is increasingly non-optional. As coding workflows move from single-agent to team-based coordination, memory systems that assume serial sessions become a bottleneck. Cortex's git-backed architecture handles this naturally, with the acknowledged caveat that heavy concurrent writes create contention that requires care.

Managed alternatives remain strong choices where integration speed and platform convenience dominate, or where semantic retrieval quality outweighs portability concerns. The right answer depends on which constraints are binding -- and that answer should be made with eyes open on the tradeoffs documented here.
