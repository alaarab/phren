Sharing & Teams

Sync phren knowledge across machines, manage access with RBAC, and collaborate across projects.

Profiles #

A profile defines which projects a machine should have access to. Profiles live as YAML files in ~/.phren/profiles/.

# ~/.phren/profiles/work.yaml
name: work
projects:
  - global
  - web-app
  - api-server
  - shared-lib

Each profile lists the project directories that should be synced and linked on machines assigned to that profile. The global entry is special: it contains skills and config that apply everywhere.

Manage profiles via CLI:

phren config profiles                              # List all profiles
phren shell                                         # Use :profile commands
:profile add-project work my-new-project            # Add a project
:profile remove-project work old-project            # Remove a project

Machines #

The machines.yaml file maps hostnames to profiles. When phren link runs on a new machine, it prompts you to register and picks a profile.

# ~/.phren/machines.yaml
work-desktop: work
home-laptop: personal
ci-server: minimal

Each machine stores its identity in ~/.phren/.machine-id. On first run, phren link asks for a machine name and profile, then writes both files.

Manage machines via CLI:

phren config machines                               # List registered machines
phren link --register                               # Re-register this machine
phren link --profile work                           # Override profile for this run
:machine map ci-runner minimal                      # Map a machine in shell

Syncing via Git #

The ~/.phren directory is a git repository. Syncing across machines is plain git:

# Push knowledge from this machine
cd ~/.phren && git add -A && git commit -m "sync" && git push

# Pull on another machine
cd ~/.phren && git pull --ff-only

phren automates this through hooks:

  • The SessionStart hook runs git pull on ~/.phren at the beginning of every session
  • The Stop hook auto-commits and pushes ~/.phren changes after every response

This means findings, tasks, and skills sync automatically as you work. No manual push/pull needed.

Sparse Checkout

phren link configures git sparse checkout so each machine only downloads the projects in its profile. This keeps large shared repos fast:

# Always included (regardless of profile)
profiles/
machines.yaml
global/
link.sh

# Plus each project in the active profile
web-app/
api-server/

Linking #

phren link is the orchestrator that wires everything together on a machine. It runs automatically during phren init and can be re-run at any time.

What it does:

  1. Identify machine — reads .machine-id, looks up profile in machines.yaml
  2. Sparse checkout — configures git to only fetch profile projects
  3. Symlink knowledge — links CLAUDE.md, FINDINGS.md, REFERENCE.md, and skills from ~/.phren/<project>/ into each project directory on disk
  4. Configure MCP — patches settings for Claude, VS Code, Cursor, Copilot, and Codex
  5. Register hooks — sets up lifecycle hooks for all detected AI tools
  6. Write context — generates ~/.phren-context.md with machine, profile, and project info
  7. Rebuild memory — updates Claude's memory files with project pointers

Task-mode linking focuses the context file on a specific workflow:

phren link --task debugging    # Include LEARNINGS.md from each project
phren link --task planning     # Include summaries + tasks
phren link --task clean        # Minimal one-line context

Consolidation #

As findings accumulate, phren detects when they need consolidation and prompts you to run the /phren-consolidate skill.

Thresholds:

  • 25 entries since the last <!-- consolidated: YYYY-MM-DD --> marker
  • Or 60 days + 10 entries since last consolidation

Consolidation archives old entries in <details> blocks, writes a date marker, and — critically — promotes patterns that appear in 2+ projects to the global scope. This is how cross-project knowledge emerges.

# Check consolidation status
phren consolidation-status
phren consolidation-status my-project

# Run consolidation
phren maintain consolidate my-project

Fragment Graph #

Fragments are named concepts that connect findings across projects. When you link a finding to a fragment, it creates a node in the knowledge graph.

# View the graph
phren graph
phren graph --project web-app

# Link a finding to a fragment
phren graph link web-app "Turbo builds avoid dist races" "build-system"

# Find cross-project fragments (MCP)
cross_project_fragments()

The cross_project_fragments MCP tool finds fragments shared across multiple projects, surfacing patterns like "both web-app and api-server have findings about caching."

Cross-Project Search #

The search_knowledge MCP tool searches across all indexed projects by default. You can scope to a single project with the project parameter.

# Search everything
search_knowledge(query: "build cache")

# Search one project
search_knowledge(query: "build cache", project: "web-app")

# CLI search
phren search "build cache"
phren search "build cache" --project web-app

Cross-project findings from global/FINDINGS.md (promoted during consolidation) are included in all searches and receive a relevance decay based on PHREN_CROSS_PROJECT_DECAY_DAYS (default: 30 days).

RBAC Roles #

phren supports four role levels for access control. The actor is identified by PHREN_ACTOR (defaults to $USER).

RolePermissions
admin Full access. Can modify governance config, access control, workflow policies, and all project data. Can approve/reject review queue items.
maintainer Can add/edit/remove findings, tasks, and skills. Can run consolidation and pruning. Cannot modify governance or access control.
contributor Can add findings and tasks. Cannot remove or edit existing entries. Cannot run maintenance operations.
viewer Read-only access. Can search, browse, and inspect all data. Cannot make any changes.

If no access control file exists, all users have full access (equivalent to admin).

Configuration #

Access control is configured in ~/.phren/.governance/access-control.json:

{
  "admins": ["alice", "bob"],
  "maintainers": ["charlie"],
  "contributors": ["dev-intern"],
  "viewers": []
}

Manage access via CLI:

phren config access get              # Show current access control
phren config access set admins alice bob
phren config access set contributors dev-intern

The actor identity is set by the PHREN_ACTOR environment variable. If unset, it defaults to $USER.

Audit Log #

All mutations are recorded in ~/.phren/.runtime/audit.log. Each entry includes the actor, action, timestamp, and relevant parameters.

Actions logged include: finding additions/removals, task mutations, consolidation runs, governance changes, and review queue operations.

Team Workflows #

For teams sharing a phren repository:

  1. Create a shared git remote for ~/.phren
  2. Define profiles per team role (e.g. frontend.yaml, backend.yaml, devops.yaml)
  3. Register each machine with phren link --register
  4. Set up RBAC so team leads are admins, developers are maintainers or contributors

The typical flow:

  • Developers work in their projects; findings and tasks auto-sync via the Stop hook
  • Consolidation runs promote shared patterns to global/FINDINGS.md
  • The review queue (phren review) surfaces stale or conflicting findings
  • Admins/maintainers prune and resolve via phren maintain prune and phren maintain govern

Example: Two-Person Team

# Alice's laptop
phren init
phren link --register   # name: alice-mbp, profile: fullstack

# Bob's desktop
git clone <phren-remote> ~/.phren
phren link --register   # name: bob-desktop, profile: fullstack

# Both work independently; Stop hook syncs after each response
# Weekly: run /phren-consolidate to merge insights

Project Ownership #

Each project has an ownership mode that controls how phren manages its files:

ModeBehavior
phren-managed Default. phren symlinks CLAUDE.md, FINDINGS.md, skills into the project directory. Full sync.
repo-owned The project's repo owns its own CLAUDE.md. phren stores findings and tasks but does not symlink files into the project directory.
read-only phren indexes and searches the project but does not write to it. 
phren projects configure my-project --ownership=repo-owned
phren config project-ownership repo-owned   # Set default for new projects
