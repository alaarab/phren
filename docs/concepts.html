Concepts

The building blocks of how phren stores, ages, and surfaces project knowledge.

Findings #

A finding is a single unit of knowledge stored in FINDINGS.md. Findings are the core primitive of phren's memory. Each one carries a type tag, a lifecycle state, and a confidence score that decays over time.

Tip

If you write a finding without a type prefix, phren auto-tags it. "We decided..." becomes [decision], "Watch out for..." becomes [pitfall], "Pattern:..." becomes [pattern].

Finding Types #

Phren recognises seven finding types:

[decision] [pattern] [pitfall] [observation] [tradeoff] [architecture] [bug]

[decision] never decays -- it represents a permanent architectural choice. [observation] expires after 14 days. All other types follow the default decay curve.

Citations: Findings can carry optional citations linking them to source code. Citations include file path, line number, repo, and commit hash. Findings with valid citations rank higher in retrieval; findings with missing citations receive a 0.8x penalty.

Adding findings:

# CLI
phren add-finding my-app "[pitfall] Auth middleware runs before rate limiting — always check order."

# MCP tool (inside an agent session)
add_finding("my-app", "[decision] Use Zod for API validation. Better TS inference than Yup.", {
  file: "src/api/schema.ts",
  line: 12
})

Trust & Decay #

Every finding starts with full confidence (1.0). As the finding ages, confidence decays through a stepped schedule:

AgeMultiplierEffect
0 -- 30 days1.0Full confidence
30 -- 60 days0.9Slightly reduced
60 -- 90 days0.75Moderate reduction
90 -- 120 days0.5Low confidence
120+ dayssuppressedExcluded from retrieval
Confidence Days

Findings that are repeatedly confirmed or frequently injected into productive sessions accumulate a decay-resistance boost and decay 3x slower than the default curve. Findings with missing citations receive an additional 0.8x penalty on top of their age-based score.

To prevent a finding from ever decaying, pin it to truths.md using pin_memory. Pinned memories are always injected and never pruned.

Important

Decay values are fully configurable via phren config policy or the retention-policy.json governance file. Adjust TTLs, decay multipliers, and prune thresholds to match your project's pace.

Consolidation #

As FINDINGS.md grows, phren triggers consolidation to keep the knowledge base clean. Consolidation deduplicates entries, archives older ones into collapsed <details> blocks, and promotes patterns that appear in 2+ projects up to global/FINDINGS.md.

Automatic triggers:

  • 25 entries added since the last consolidation marker, or
  • 60 days have passed with at least 10 entries.

Phren fires a one-time notice per session when the threshold is met. Run it manually:

phren maintain consolidate my-app
# Preview without writing:
phren maintain consolidate my-app --dry-run
Tip

Archived entries (inside <details> blocks) are stripped from the FTS5 index. Old findings never pollute search results.

Fragment Knowledge Graph #

A fragment is a named concept or topic that links across findings and docs. Fragments let you trace a topic across projects. For example, a fragment called auth-flow could link findings from two projects and a reference doc.

Working with fragments:

# Find fragments by name
search_fragments("rate-limiting")

# Link a finding to a fragment
link_findings("my-app", "The auth middleware...", "middleware-order", "mentions")

# See the full graph
phren graph --project my-app

Cross-Project Links #

Use cross_project_fragments() to discover which fragments span multiple projects. These shared fragments often reveal architectural patterns, shared dependencies, or recurring pitfalls across your codebase.

Note

Fragment relations include mentions, implements, contradicts, and extends. The relation type affects how fragments are weighted in search results.

Session Continuity #

Sessions track continuity across agent restarts. When a session starts, phren provides everything the agent needs to resume where it left off:

  • Prior session's summary
  • Recent findings added since last session
  • The active task and its checkpoint hints
  • Any pending review queue items
sequenceDiagram
    participant Agent
    participant Phren
    participant Files as ~/.phren

    Agent->>Phren: session_start("my-app")
    Phren->>Files: Read last session summary
    Phren->>Files: Read recent findings
    Phren->>Files: Read task checkpoints
    Phren-->>Agent: Summary + findings + active task + hints

    Note over Agent,Phren: Agent works...

    Agent->>Phren: add_finding(...)
    Agent->>Phren: complete_task(...)

    Agent->>Phren: session_end("Implemented rate limiting")
    Phren->>Files: Write session summary
    Phren->>Files: Save task checkpoints

For Claude with native hooks, session lifecycle runs automatically. For other agents, call session_start and session_end explicitly:

# At the start of a session (MCP):
session_start("my-app")

# At the end:
session_end("Implemented rate limiting middleware, linked to auth flow.")

Task Checkpoints #

Task checkpoints record the state of in-progress work so agents can resume seamlessly. Each checkpoint captures:

  • Task ID -- which task was being worked on
  • Edited files -- which files were modified
  • Failing tests -- any tests that were still failing
  • Next-step hints -- what the agent planned to do next

Completing a task automatically clears its checkpoint file. Checkpoints are stored in .sessions/ and are ephemeral -- they exist only to bridge agent restarts.

Warning

Session summaries are critical for continuity. If an agent crashes without calling session_end, the next session starts with stale context. The Stop hook mitigates this by auto-saving partial state.
