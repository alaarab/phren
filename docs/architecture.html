System Overview

How project memory flows through the system, from user prompt to repo-backed state and back into bounded retrieval.

Phren is a knowledge layer for AI agents. It sits between the agent (Claude, Copilot, Cursor, Codex) and a local file store, intercepting lifecycle events to inject context and persist state. The current public surface is 66 MCP tools across 12 modules.

▼

Install Modes

Phren has two install modes, rooted by phren.root.yaml:

ModeRootDescription
shared ~/.phren Default personal memory with profiles, machine mappings, user-scoped MCP config, and full Claude lifecycle hooks
project-local <repo>/.phren Repo-owned memory with one primary project, workspace-managed git, and workspace MCP wiring

Path Resolution Order

  1. Explicit CLI path argument
  2. PHREN_PATH environment variable
  3. Nearest ancestor .phren containing phren.root.yaml
  4. Shared root at ~/.phren only if it contains phren.root.yaml

End-to-End Runtime Loop

Every prompt triggers a five-stage pipeline: trigger, retrieve, govern, persist, repeat.

1
Session Lifecycle Trigger
Claude fires SessionStart / UserPromptSubmit / Stop natively. Copilot, Cursor, and Codex use a session wrapper plus tool hook config.
2
Retrieval
Keyword extraction and synonym expansion feed into lexical-first FTS5 retrieval. When lexical confidence is low, an optional vector fallback fires. Results go through recency-aware and task-aware reranking.
3
Governance
Citation checks, confidence decay, and policy gates run against retrieved items. Stale or low-confidence items are filtered out or queued for review.
4
Persistence
MCP tool calls write to markdown and JSON files. The Stop lifecycle hook persists changes via git add, commit, and push through a sync worker.
5
Next Prompt
Retrieval reads the newly persisted state. The loop repeats.

Three paths on every prompt cycle: retrieval (FTS5 search + snippet injection), governance (citation validity + confidence decay), and persistence (MCP writes + Stop hook git commit/push).

Hook and Integration Model

How phren wires into each agent's lifecycle.

Claude uses native lifecycle hooks registered in ~/.claude/settings.json:

HookHandlerWhat it does
SessionStart hook-session-start Git pull on ~/.phren, health self-check, project context injection
UserPromptSubmit hook-prompt Keyword extraction, FTS5 search, context snippet injection, consolidation threshold check
Stop hook-stop Git add, commit, and push ~/.phren changes after every response

Copilot CLI, Cursor, and Codex use two layers:

  • Generated per-tool hook config files that map lifecycle events to phren handlers
  • Session wrapper binaries in ~/.local/bin/ that enforce lifecycle behavior around each tool invocation

This gives Claude full native lifecycle parity while keeping other tools synchronized through wrappers and config.

Findings: Lifecycle, Provenance, Impact

Findings carry rich state: provenance, lifecycle, and impact scoring.

Provenance Sources

Each finding records its origin via add_finding.source:

human agent hook extract consolidation unknown

Lifecycle States

Lifecycle state is stored inline in the finding text:

active superseded contradicted retracted stale invalid_citation

Contradiction and supersession are handled by dedicated lifecycle tools (supersede_finding, retract_finding, resolve_contradiction).

Impact Scoring Pipeline

  1. Injected findings are logged by ID when surfaced in context
  2. Session completion checks for task completion in that session
  3. Completed outcomes update impact logs
  4. High-impact findings get boosted in future retrieval

Session Continuity

Three tools maintain continuity across sessions.

ToolWhat it does
session_start Returns prior summary, recent findings, and checkpoint hints
session_end Writes summary and task checkpoint snapshots
session_history Lists sessions or returns detailed session artifacts

Task checkpoints include task ID/text, edited files, failing tests, and resume hints (lastAttempt and nextStep). Completed tasks clear their checkpoint files.

Skill Resolution System

Deterministic and policy-aware skill lookup.

  • Precedence: project scope overrides global scope for the same skill name
  • Alias collisions: colliding commands/aliases are marked unregistered
  • Visibility gating: disabled skills stay on disk but are hidden from active agent mirrors

Governance Identity and RBAC

Identity and authorization flow for access control.

Identity Resolution

  • Actor identity from PHREN_ACTOR (in trusted/test contexts) or OS user identity
  • Shared role policy from .governance/access-control.json
  • Local fallback/augmentation from .runtime/access-control.local.json

RBAC Roles

admin maintainer contributor viewer

Write, policy, and delete operations are checked against RBAC before mutation.

Web UI Security Model

The memory web UI (phren web-ui) is hardened by default.

  • Binds only to loopback (127.0.0.1)
  • Issues random per-run auth token (bearer/query/body)
  • Requires CSRF token for mutating routes
  • CSRF tokens are single-use with TTL
  • Sets CSP and anti-framing headers (Content-Security-Policy, X-Frame-Options, X-Content-Type-Options)
Mutating endpoints require both auth and CSRF. Read-only endpoints require auth only.

Retrieval and Context Efficiency

Bounded context usage keeps memory growth decoupled from prompt size.

  • Lexical-first FTS5 path handles most queries without vector overhead
  • Vector fallback runs only when lexical confidence is weak
  • Progressive disclosure keeps injection bounded via PHREN_CONTEXT_TOKEN_BUDGET
  • Task APIs support summary, pagination, and single-item fetch patterns
Memory growth and prompt context growth are decoupled. The token budget controls how much context enters each prompt, regardless of how large the knowledge store grows.
