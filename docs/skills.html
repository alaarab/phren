Skills Guide

Reusable instructions that agents invoke as slash commands. Write once, use everywhere.

What Are Skills #

A skill is a markdown file with YAML frontmatter that defines a reusable procedure. When you type /skill-name in an agent session, the agent reads the skill file and follows its instructions.

Skills encode workflows, checklists, and multi-step procedures that survive across sessions and machines. Instead of re-explaining a process every time, write it once as a skill and invoke it by name.

  • Portable — skills live in ~/.phren and sync across machines via git
  • Scoped — global skills apply everywhere; project skills override globals with the same name
  • Versioned — skills are markdown in a git repo, so you get full history and rollback
  • Agent-agnostic — any agent with MCP access can read and execute skills

Anatomy of a Skill #

A skill file has two parts: YAML frontmatter with metadata, and a markdown body with the instructions the agent follows.

---
name: deploy-staging
command: /deploy-staging
description: Deploy the current branch to staging
tags: [devops, deploy]
---

# Deploy to Staging

1. Run the test suite: `npm test`
2. Build the production bundle: `npm run build`
3. Deploy to staging: `./scripts/deploy.sh staging`
4. Verify the deployment at https://staging.example.com
5. Report the deployment status.

Creating Skills #

Create a .md file with YAML frontmatter and place it in a skills/ directory. That is all it takes.

Frontmatter Format

The YAML frontmatter defines metadata that phren uses for resolution, discovery, and the command system.

Field Required Description
name Yes Unique identifier for the skill. Used for resolution and display.
command No The slash command that invokes this skill (e.g. /deploy). Defaults to /name.
description No One-line description shown in skill listings.
tags No Array of tags for categorization and discovery.
aliases No Alternative command names that also invoke this skill.

Skill Body

The body is regular markdown. Write it as instructions the agent should follow when the skill is invoked:

  • Step-by-step procedures
  • Code blocks with commands to run
  • Conditional logic ("if X, then do Y")
  • References to other tools or skills
  • Validation checks ("verify that...")

Write for the agent, not for humans. Be explicit about what to do, what to check, and what to report back. Avoid ambiguity.

File Structure

Skill files are plain .md files placed in a skills/ directory:

# Global skill
~/.phren/global/skills/deploy-staging.md

# Project skill
~/.phren/my-project/skills/run-tests.md

You can also create skills via CLI or MCP:

# CLI
phren skills add my-project ./my-skill.md

# MCP tool
write_skill("deploy-staging", "---\nname: deploy-staging\n...", "global")

Global vs Project Skills #

Skills exist in two scopes. Project always wins.

  • Global skills — live in ~/.phren/global/skills/. Available in every project. Good for personal workflow commands that apply everywhere.
  • Project skills — live in ~/.phren/<project>/skills/. Only available when working in that project. Good for project-specific procedures.

Examples of global skills

  • /phren-sync — pull phren to a new machine or push config changes
  • /phren-init — scaffold a new project with summary, CLAUDE.md, tasks
  • /phren-discover — research what is missing in a project
  • /phren-consolidate — find patterns across all FINDINGS.md files

Examples of project skills

  • /deploy-staging — deploy procedure specific to one project
  • /run-e2e — end-to-end test workflow for a specific app
  • /release — release checklist for a library

Resolution Order #

When an agent resolves a skill name or command, phren checks in this order:

  1. Project skills — ~/.phren/<project>/skills/
  2. Global skills — ~/.phren/global/skills/

If a project skill has the same name as a global skill, the project skill wins. The global skill is shadowed, not deleted.

Collision Handling

When two skills at the same scope share an alias or command name, phren flags the collision and leaves the conflicting command unregistered. The generated artifacts reflect this:

  • .claude/skill-manifest.json — full resolution map with collision warnings
  • .claude/skill-commands.json — registered commands (excluding collisions)

Diagnose resolution issues:

phren skills resolve my-project   # Print resolved skill manifest
phren skills doctor my-project    # Diagnose skill visibility issues

Disabled skills stay on disk but are excluded from resolution, mirrors, and command generation. They do not shadow global skills of the same name.

Built-in Skills #

These skills ship with phren and are copied to ~/.phren/global/skills/ on init:

Command What it does
/phren-sync Pull phren to a new machine or push config changes back to the repo
/phren-init Scaffold a new project with summary, CLAUDE.md, tasks
/phren-discover Research what is missing in a project and surface gaps and opportunities
/phren-consolidate Find patterns across all project FINDINGS.md files

You can customize these — your edits persist across updates since init does not overwrite existing files.

The /command System #

When you type /skill-name in an agent session, the agent uses the read_skill MCP tool (or reads the file directly) to load the skill content, then follows the instructions in the body.

  1. You type /deploy-staging in your agent
  2. The agent resolves deploy-staging using the skill manifest
  3. The agent reads the full skill file (frontmatter + body)
  4. The agent follows the instructions step by step

Skills can accept arguments. Pass them after the command name:

/phren-init my-new-project
/deploy-staging --branch feature/auth

The agent receives the full invocation string and interprets arguments based on the skill body instructions.

Managing Skills #

CLI commands

phren skills list                            # List all installed skills
phren skills add my-project ./skill.md       # Add a skill to a project
phren skills show deploy --project my-project # Show skill content
phren skills edit deploy --project my-project # Edit in $EDITOR
phren skills resolve my-project               # Print resolved manifest
phren skills doctor my-project                # Diagnose visibility
phren skills sync my-project                  # Regenerate skill mirror
phren skills enable my-project deploy         # Re-enable a disabled skill
phren skills disable my-project deploy        # Disable without deleting
phren skills remove my-project deploy         # Delete permanently

Detect untracked skills that exist in ~/.claude/skills/ but are not registered:

phren detect-skills           # Find untracked skills
phren detect-skills --import  # Find and import them

MCP tools

From inside an agent session:

list_skills()                              # All installed skills
list_skills("my-project")                  # Project-specific skills
read_skill("deploy-staging")               # Full content + frontmatter
write_skill("deploy", content, "global")   # Create or update
enable_skill("deploy", "my-project")       # Re-enable
disable_skill("deploy", "my-project")      # Disable
remove_skill("deploy", "my-project")       # Delete

Shell view

In the interactive shell, press s to open the Skills view. It shows all global and project-local skills with their enabled/disabled state. Press t to toggle a skill's enabled state.
