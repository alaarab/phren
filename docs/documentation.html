Getting Started

Install phren and get context injection running in your first session.

What phren init does

Running npx @phren/cli init does the following in order:

  1. Creates ~/.phren/ and initializes it as a git repo.
  2. Detects which agents are installed: Claude Code, VS Code Copilot, Cursor, Codex CLI.
  3. Registers the phren MCP server in each agent's config file.
  4. Wires up lifecycle hooks (SessionStart, UserPromptSubmit, Stop) for Claude. Generates wrapper binaries in ~/.local/bin/ for the other agents.
  5. Creates a machines.yaml entry for the current hostname.
  6. Copies starter templates into ~/.phren/ (CLAUDE.md, skills, governance defaults).
One command to install: npm install -g @phren/cli && phren init

Re-running phren init is safe. It updates MCP config and hooks without touching existing data.

First 10 minutes

After init, open any project directory in Claude Code (or your preferred agent) and start a prompt. Here is what happens:

  1. SessionStart fires: phren pulls the latest ~/.phren changes and runs a health self-check.
  2. UserPromptSubmit fires: phren extracts keywords from your prompt, searches its FTS5 index, and prepends relevant context snippets to what the agent sees.
  3. You work. When you call MCP tools like add_finding or add_task, phren writes to markdown files under ~/.phren/.
  4. Stop fires after every agent response: phren commits and pushes ~/.phren to its remote (if one is configured).

On your second session, phren injects context from the first. Findings, tasks, and session summaries all carry over.

Run phren verify after init to confirm hooks and MCP are wired correctly.

Adding projects

To enroll a repo or directory as a phren project:

cd ~/my-project
phren add .

This creates ~/.phren/my-project/ with CLAUDE.md, tasks.md, FINDINGS.md, and summary.md. From that point on, phren injects context from this project whenever your working directory is inside it.

To update or remove a project:

phren projects list
phren projects configure my-project --ownership=phren-managed
phren projects remove my-project

To update phren itself:

phren update
# or, to also refresh bundled starter templates:
phren update --refresh-starter

Concepts

The building blocks of how phren stores, ages, and surfaces project knowledge.

Findings

A finding is a single unit of knowledge stored in FINDINGS.md. Findings are the core unit of phren's memory. Each one has a type tag, a lifecycle state, and a confidence score that decays over time.

Eleven finding types:

[decision] [pattern] [pitfall] [observation] [bug] [workaround] [tooling] [context] [anti-pattern] [tradeoff] [architecture]

[decision] never decays. [observation] expires in 14 days. The rest follow the default decay curve.

Example:

phren add-finding my-app "[pitfall] The auth middleware runs before rate limiting. Always check order in middleware stack."

Or via MCP tool in an agent session:

add_finding("my-app", "[decision] Use Zod for all API input validation. Chosen over Yup for better TS inference.", {
  file: "src/api/schema.ts",
  line: 12
})

Finding lifecycle states: active, superseded, contradicted, retracted, stale, invalid_citation. Lifecycle tools (supersede_finding, retract_finding, resolve_contradiction) update inline metadata. Superseded findings score 0.25x in retrieval, retracted 0.1x, contradicted 0.4x. Superseded and retracted findings are stripped from the FTS index entirely.

Auto-tagging: If you write a finding without a type prefix, phren tries to detect the right type from the content. "We decided..." becomes [decision], "Watch out for..." becomes [pitfall], "Pattern:..." becomes [pattern], and so on.

Fragments

A fragment is a named concept or topic that links across findings and docs. Fragments let you trace a topic across projects. For example, a fragment called auth-flow could link findings from two projects and a reference doc.

Use search_fragments to find fragments by name, link_findings to attach a finding to a fragment, and cross_project_fragments to see which fragments span multiple projects.

search_fragments("rate-limiting")
link_findings("my-app", "The auth middleware...", "middleware-order", "mentions")

Trust and decay

Every finding starts with full confidence (1.0). Confidence decays as the finding ages. The default schedule:

AgeMultiplierEffect
0–30 days1.0Full confidence
30–60 days0.85Slightly reduced
60–90 days0.65Moderate reduction
90–120 days0.45Low confidence
120+ days0.0Expired, prunable

Findings that are repeatedly confirmed or frequently injected into productive sessions accumulate a decay-resistance boost. They decay 3x slower than the default curve.

To prevent a finding from ever decaying, pin it to truths.md using pin_memory. Pinned memories are always injected and never pruned.

Decay values are configurable via phren config policy or the retention-policy.json governance file.

Consolidation

As FINDINGS.md grows, phren triggers consolidation. Consolidation deduplicates entries, archives older ones into collapsed <details> blocks, and promotes patterns that appear in 2+ projects up to global/FINDINGS.md.

Consolidation fires automatically when:

  • 25 entries have been added since the last consolidation marker, or
  • 60 days have passed and there are at least 10 entries.

Phren fires a one-time notice per session when the threshold is met. You can also run consolidation manually:

phren maintain consolidate my-app
# Preview without writing:
phren maintain consolidate my-app --dry-run

Archived entries (inside <details> blocks) are stripped from the FTS index, so old findings do not pollute search results.

Sessions

Sessions track continuity across agent restarts. At session_start, phren returns the prior session's summary, recent findings, the active task, and any task checkpoint hints so the agent knows where it left off. At session_end, phren writes a summary and saves task checkpoints.

Task checkpoints record task ID, edited files, failing tests, and next-step hints. Completing a task clears its checkpoint file.

For Claude with native hooks, session lifecycle runs automatically. For other agents, call session_start and session_end explicitly.

# At the start of a session (MCP):
session_start("my-app")

# At the end:
session_end("Implemented rate limiting middleware, linked to auth flow.")

Skills

Skills are slash commands defined as markdown files with YAML frontmatter. They live in ~/.phren/global/skills/ (available everywhere) or ~/.phren/<project>/skills/ (project-local, overrides global with the same name).

When an agent detects a skill, it reads the markdown body as instructions and executes the described workflow. Phren generates .claude/skill-manifest.json and .claude/skill-commands.json so agents can discover available skills.

Example skill frontmatter:

---
name: phren-sync
description: Pull phren to a new machine or push config changes back.
triggers: ["/phren-sync", "/sync"]
---

# phren-sync

Pull the latest ~/.phren changes: `cd ~/.phren && git pull --rebase`
Then run `phren verify` to confirm everything is wired.

Hooks

Hooks are shell commands that phren registers in each agent's config. For Claude, they run at four lifecycle points: SessionStart, UserPromptSubmit, Stop, and PostToolUse. The hooks do context injection, git sync, and persistence automatically, without any prompt required.

Hooks can be toggled globally or per tool:

phren hooks-mode off         # disable all hooks
phren hooks enable cursor    # re-enable just cursor hooks

Skills

Custom slash commands for your agent workflows.

A skill is a markdown file that defines a slash command. When an agent encounters the trigger (e.g. /deploy-check), it reads the skill body as instructions and follows them.

Where skills live

LocationScope
~/.phren/global/skills/Available in all projects
~/.phren/<project>/skills/Available only in that project. Overrides a global skill with the same name.

Skill file format

Skills are markdown files with YAML frontmatter:

---
name: review-pr
description: Review the current branch diff for issues before pushing.
triggers: ["/review-pr", "/review"]
enabled: true
---

# review-pr

1. Run `git diff main...HEAD` and read the output.
2. Check for hardcoded secrets, missing tests, and TODO comments.
3. Summarize findings in a short list.
4. Add any non-obvious issues as phren findings: `add_finding("project", "[pitfall] ...")`

Managing skills

# List all skills
phren skills list

# Create or update a skill via MCP:
write_skill("review-pr", "<content>", "global")

# Disable without deleting (stays on disk, hidden from agent):
phren skills disable global review-pr

# Re-enable:
phren skills enable global review-pr

Phren generates .claude/skill-manifest.json and .claude/skill-commands.json so agents automatically discover available skills. A project-local skill with the same name as a global skill takes precedence.

Alias collisions: If two skills define conflicting command triggers, phren marks the conflicting one as unregistered in the generated manifest. Resolve by renaming one of the triggers.

Hooks

What fires, when, and how to add your own integrations.

Phren hooks are shell commands registered in each AI agent's config file. They run automatically at lifecycle events: injecting context before each prompt, committing changes after each response, and syncing state at session boundaries. Hooks fire independently per tool — Claude, Codex, Copilot CLI, and Cursor each use their own config format.

Hooks run as node <entry> <command> when a local install exists, which avoids the cold-start delay of npx.

SessionStart

Fires when an agent session begins, before the first prompt.

hook-session-start does:

  1. Runs git pull --rebase on ~/.phren to sync across machines
  2. Cleans up stale session markers older than 24 hours

hook-context also runs on SessionStart and detects the current project from cwd, then injects its CLAUDE.md and summary.md into the session context.

UserPromptSubmit

Fires before each user prompt is processed. This is the main context injection pipeline.

hook-prompt does, in order:

  1. Reads the prompt JSON from stdin ({ prompt, cwd, session_id })
  2. Checks global and per-tool hook preferences; exits silently if disabled
  3. Extracts keywords from the prompt with stop-word filtering
  4. Tracks topic frequency within the session for momentum boosting
  5. Builds an FTS5 query with synonym expansion
  6. Searches the in-memory SQLite index, with TF-IDF hybrid fallback
  7. Applies the trust filter: strips entries below TTL, applies confidence decay
  8. Ranks results with intent detection, git context, and project glob boosting
  9. Selects snippets within the token budget (PHREN_CONTEXT_TOKEN_BUDGET, default 550 tokens)
  10. Deduplicates snippets: if the same bullet appears in project and global files, it is shown once
  11. Checks the consolidation threshold once per session; injects a <phren-notice> if consolidation is recommended
  12. Writes formatted context to stdout; the agent prepends it to the prompt
Session momentum: Topics queried 3+ times in a session get a 30% larger token budget slice, so frequently-relevant context stays well-represented as work progresses.

Stop

Fires after each agent response, including subagent responses.

hook-stop does:

  1. Runs git add -- *.md *.json *.yaml *.yml *.jsonl *.txt in ~/.phren
  2. Commits with an auto-generated message listing changed files and projects
  3. Pushes to the configured remote if one exists
  4. Fires the pre-save custom hook before staging and post-save after pushing

The Stop hook does not run consolidation to avoid nested hook invocations.

PostToolUse

Fires after an agent tool call completes. Claude Code only.

hook-tool reads the tool result from stdin and optionally extracts findings from tool output based on proactivity settings. It updates session metrics to track tool usage within the session.

Integration per tool

ToolConfig fileIntegration method
Claude Code ~/.claude/settings.json Native lifecycle hooks + MCP (SessionStart, UserPromptSubmit, Stop, PostToolUse)
GitHub Copilot CLI ~/.github/hooks/phren.json MCP + generated hook config + session wrapper binary in ~/.local/bin
Cursor ~/.cursor/hooks.json MCP + generated hook config + session wrapper binary
Codex CLI ~/.phren/codex.json MCP + generated hook config via npx (no stable local binary path)

For Copilot, Cursor, and Codex, phren init installs a thin shell wrapper at ~/.local/bin/<tool> that runs hook-session-start before the real binary and hook-stop after it exits. This ensures lifecycle behavior fires even when the tool's own hook support is incomplete.

Toggling hooks

# Disable all hooks globally
phren hooks-mode off

# Disable hooks only for one tool
phren hooks disable cursor

# Re-enable
phren hooks enable cursor
phren hooks-mode on

When MCP is off but hooks are on, context injection still happens via hooks. When hooks are off, hook commands exit immediately with code 0.

Per-project overrides

Individual lifecycle events can be enabled or disabled per project without affecting the global setting. Useful for projects with large or noisy knowledge bases where you want to suppress automatic context injection.

Config file: ~/.phren/<project>/phren.project.yaml

hooks:
  enabled: true           # base override for this project (null = inherit global)
  UserPromptSubmit: true  # per-event overrides
  Stop: true
  SessionStart: true
  PostToolUse: false

Or via MCP:

# Disable UserPromptSubmit injection for one project only
toggle_hooks({ enabled: false, project: "my-app", event: "UserPromptSubmit" })

# Disable all hooks for a project
toggle_hooks({ enabled: false, project: "my-app" })

Custom hooks

Custom hooks let you run shell commands or POST webhooks on phren internal events. They are stored in ~/.phren/.governance/install-preferences.json.

Available events

EventWhen it fires
pre-saveBefore push_changes stages files
post-saveAfter push_changes pushes to remote
post-searchAfter search_knowledge returns results
pre-findingBefore a finding is written to FINDINGS.md
post-findingAfter a finding is written
pre-indexBefore FTS5 index rebuild
post-indexAfter FTS5 index rebuild
post-session-endAfter session_end completes
post-consolidateAfter FINDINGS.md consolidation runs

Shell command hooks

Shell commands run via sh -c with PHREN_PATH, PHREN_HOOK_EVENT, and per-event extras like PHREN_PROJECT and PHREN_QUERY set in the environment.

# Add a command hook via MCP:
add_custom_hook({ event: "post-finding", command: "notify-slack finding-saved" })

# Remove it:
remove_custom_hook({ event: "post-finding" })

Restrictions: no shell metacharacters (` $ ( ) { } & | ; < >), no eval or source, max 1000 chars, default timeout 5000ms.

Webhook hooks

Webhook hooks POST to a URL asynchronously (fire-and-forget). The request body is:

{
  "event": "post-finding",
  "env": { "PHREN_PROJECT": "backend" },
  "timestamp": "2026-03-14T10:00:00.000Z"
}

If a secret is provided, the request includes an X-Phren-Signature: sha256=<hex> header for HMAC-SHA256 verification. Private/loopback addresses (localhost, 127.x, 192.168.x, etc.) are rejected.

add_custom_hook({
  event: "post-session-end",
  webhook: "https://example.com/phren-events",
  secret: "my-signing-secret"
})

Troubleshooting

# Check recent hook errors
phren doctor
cat ~/.phren/.runtime/hook-errors.log

# Via MCP:
list_hook_errors({ limit: 50 })

# Enable debug logging (verbose stderr + debug.log)
PHREN_DEBUG=1 phren hook-prompt

# Re-link hooks if settings files are missing or malformed
phren doctor --fix

The hook error log rotates at 200 KB, keeping the last 1000 lines. The default hook timeout is 14 seconds (PHREN_HOOK_TIMEOUT_MS). If hooks time out frequently, check whether ~/.phren is on a slow or network filesystem.

Multi-machine

How phren keeps knowledge in sync across machines using profiles and git.

~/.phren is a git repo. Phren's Stop hook commits and pushes after every response. The SessionStart hook pulls before every session. This means your findings, tasks, and session history stay in sync across any machine that shares the same remote.

Profiles

A profile is a named set of projects. Different machines can load different profiles. For example, a work laptop might load the work profile, while a home machine loads personal.

Profile files live in ~/.phren/profiles/ as YAML files:

# ~/.phren/profiles/work.yaml
name: work
projects:
  - api-backend
  - data-pipeline
  - infra

machines.yaml

This file maps machine hostnames to profiles:

# ~/.phren/machines.yaml
machines:
  my-laptop: work
  my-desktop: personal
  ci-runner: ci

When phren starts on a machine, it reads the hostname, looks up the profile in machines.yaml, and only indexes projects in that profile.

Setting up a new machine

# On the new machine:
npm install -g @phren/cli

# Link to your existing remote (pass --remote if needed):
phren init --link

# Or run init and then pull manually:
phren init
cd ~/.phren && git remote add origin <your-remote> && git pull --rebase

# Verify:
phren verify

After setup, the Stop hook will push from the new machine, and the SessionStart hook will pull on each session start.

Manual sync

# Pull latest changes:
cd ~/.phren && git pull --rebase

# Or push uncommitted changes:
push_changes("sync: manual push from home machine")
No conflict resolution needed in most cases. Phren writes to date-stamped sections in markdown files, so concurrent changes from multiple machines rarely conflict. If they do, standard git merge applies.

Graph Engine API

The fragment graph webview exposes a small JS API for host integration — used by the VS Code extension and the phren web UI.

The graph engine is mounted into a container element and communicates back to the host via a callback. All coordinates use CSS pixel space.

mount(data)Initialize the graph with node and edge data. Call once after the container is in the DOM. data is the JSON payload from read_graph.
onNodeSelect(cb)Register a callback that fires when a node is selected by click or keyboard. Receives (node, canvasX, canvasY) where node is the full node object. Replaces any previously registered callback.
getNodeAt(x, y)Hit-test at the given CSS coordinates (relative to the canvas bounding rect). Returns the node object under the pointer, or null if none.
getCurrentNode()Returns the node ID where the phren character is currently located, or null if phren hasn't moved yet.
walkTo(nodeId)Move the phren character to a node by ID using BFS edge-walking. Animates the traversal path. Returns true if the node was found, false otherwise. Does not fire onNodeSelect.

Keyboard navigation: Arrow keys move focus between adjacent nodes. Enter selects the focused node and fires onNodeSelect. Escape clears focus.

Example:

// After the graph canvas is in the DOM:
window.phrenGraph.mount({ nodes, links, scores, topics });
window.phrenGraph.onNodeSelect(function(node, cx, cy) {
  console.log('Selected:', node.id, node.label, 'at', cx, cy);
});

// Programmatic navigation:
window.phrenGraph.walkTo("auth-flow");
